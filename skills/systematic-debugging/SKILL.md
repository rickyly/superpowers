# 系统化调试

## 概述

随手修补是浪费时间，还会制造新 bug。快速补丁只会掩盖真正的问题。

**核心原则**：动手修复之前，ALWAYS 先找到 root cause（根因）。修补表象就是失败。

**违背本流程的字面要求，就是违背调试的精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果没有完成 Phase 1，你就不能提出修复方案。

## 何时使用

适用于任何技术问题：

- 测试失败
- 生产环境的 bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**以下情形尤其要用**：

- 时间压力大时（紧急情况下靠猜尤其诱人）
- 「就改这一行」看起来很明显时
- 已经尝试过多个修复时
- 上一个修复没奏效时
- 你还没完全弄懂问题时

**以下情形不要跳过**：

- 问题看起来很简单（简单 bug 也有 root cause）
- 你在赶时间（越赶越要返工）
- 经理要求马上修好（系统化反而比乱撞更快）

## 四个阶段

你 MUST 完成每个阶段后才能进入下一个。

### Phase 1: Root Cause 调查

**在尝试任何修复之前**：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常已经包含了确切的解决方案
   - 完整读完 stack trace
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果不可复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - 有哪些改动可能导致这个问题？
   - `git diff`、近期 commit
   - 新的依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统由多个组件组成时（CI → build → signing、API → service → database）**：

   **在提出修复之前，加入诊断埋点**：
   ```
   对每一个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境变量/配置是否传递下去
     - 检查每一层的状态

   先跑一次收集证据，看它在哪里断掉
   然后分析证据，定位到失败的组件
   然后再去调查那个具体的组件
   ```

   **示例（多层系统）**：
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这会揭示**：哪一层失败了（secrets → workflow 通过，workflow → build 断掉）

5. **追踪数据流**

   **当错误出现在调用栈深处时**：

   完整的反向追踪技巧参见本目录下的 `root-cause-tracing.md`。

   **速记版**：
   - 坏值从哪里产生？
   - 是谁用坏值调用了这里？
   - 一直向上追溯，直到找到源头
   - 在源头修，而不是在症状处修

### Phase 2: Pattern 分析

**修复前先找出 pattern**：

1. **找到可用的范例**
   - 在同一代码库里找相似的、能工作的代码
   - 和坏掉那部分相似、却能跑起来的是什么？

2. **对照参考实现**
   - 如果要实现某个 pattern，完整读完参考实现
   - 不要扫读——逐行阅读
   - 完全理解这个 pattern 后再应用

3. **识别差异**
   - 能工作的代码和坏掉的代码之间有哪些不同？
   - 列出所有差异，再小也要列
   - 不要预设「这个差别不可能有影响」

4. **理解依赖**
   - 它还需要哪些组件？
   - 需要哪些设置、配置、环境？
   - 它假定了什么前提？

### Phase 3: 假设与验证

**科学方法**：

1. **提出单一假设**
   - 明确表述：「我认为 X 是 root cause，因为 Y」
   - 写下来
   - 要具体，不要含糊

2. **最小化验证**
   - 做出最小的改动来验证假设
   - 一次只动一个变量
   - 不要一次修多个东西

3. **继续之前先验证**
   - 奏效了吗？是 → Phase 4
   - 没奏效？提出新的假设
   - DON'T 在旧修复上再叠加新修复

4. **当你不知道时**
   - 说「我不明白 X」
   - 不要装懂
   - 寻求帮助
   - 做更多研究

### Phase 4: 实施

**修 root cause，不要修症状**：

1. **写一个失败的测试用例**
   - 最简单的可能复现
   - 可能的话写成自动化测试
   - 没有测试框架时，写一个一次性脚本
   - MUST 在修复前就有测试
   - 使用 `superpowers:test-driven-development` skill 来写像样的 failing test

2. **实施单一修复**
   - 针对已识别出的 root cause
   - 一次只改一处
   - 不要顺手加「既然来了就一起改」
   - 不要捆绑 refactor

3. **验证修复**
   - 测试现在通过吗？
   - 有没有弄坏其他测试？
   - 问题真的解决了吗？

4. **如果修复不起作用**
   - STOP
   - 数一数：已经尝试过几次修复？
   - 少于 3 次：回到 Phase 1，结合新信息重新分析
   - **达到或超过 3 次：STOP，开始质疑架构（见下面第 5 步）**
   - DON'T 在没讨论架构的情况下尝试第 4 次修复

5. **如果已失败 3 次以上：质疑架构**

   **提示架构问题的 pattern**：
   - 每次修复都在不同位置暴露出新的共享状态/耦合/问题
   - 修复需要「大规模 refactor」才能完成
   - 每次修复都会在别处引发新的症状

   **STOP，质疑底层假设**：
   - 这个 pattern 本身是否站得住脚？
   - 是不是「只是因为惯性才一直用它」？
   - 应该 refactor 架构，还是继续修症状？

   **在尝试更多修复前和你的人类伙伴讨论**

   这不是「假设失败」，而是「架构错了」。

## Red Flags——STOP，回到流程

如果你发现自己冒出这些想法：

- 「先快速修一下，之后再查」
- 「先改一下 X 看看行不行」
- 「一次加好几处改动，跑测试」
- 「跳过测试，我手动验证」
- 「八成是 X，我来改」
- 「虽然没完全搞懂，但这样改说不定行」
- 「Pattern 是 X，但我稍微改一下应该也行」
- 「主要问题是：[在没有调查的情况下列出一堆修复项]」
- 在追踪数据流之前就提出解决方案
- **「再试最后一次修复」（已经试过 2 次以上时）**
- **每次修复都在不同位置暴露出新的问题**

**所有这些都意味着：STOP。回到 Phase 1。**

**如果已失败 3 次以上**：质疑架构（见 Phase 4.5）

## 你的人类伙伴在暗示你搞错了

**留意这些纠正信号**：

- 「那件事没发生吗？」——你没验证就假设了
- 「这能让我们看到……吗？」——你本该先加埋点收集证据
- 「别猜了」——你在没理解问题的情况下提修复
- 「Ultrathink 一下」——该质疑底层假设，而不只是症状
- 「我们卡住了吗？」（带着挫败感）——你的路子走不通

**看到这些信号时**：STOP。回到 Phase 1。

## 常见的自我合理化

| 借口 | 真相 |
|------|------|
| 「问题很简单，不用走流程」 | 简单问题也有 root cause。流程对简单 bug 同样很快。 |
| 「情况紧急，没时间走流程」 | 系统化调试比乱猜乱试更快。 |
| 「先试一下，然后再调查」 | 第一次修复会定下基调。从一开始就做对。 |
| 「修好之后再补测试」 | 未经测试的修复不会稳定。先写测试才能证明修好了。 |
| 「一次改多个地方省时间」 | 根本分不清哪一处起了作用。还会引入新 bug。 |
| 「参考实现太长，我自己套一下 pattern 就行」 | 一知半解必然引发 bug。完整读完它。 |
| 「我看到问题了，让我来修」 | 看到症状 ≠ 理解 root cause。 |
| 「再试最后一次修复」（已失败 2 次以上时） | 3 次以上失败 = 架构有问题。质疑 pattern，别再修了。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. Root Cause** | 读错误、复现、查变更、收集证据 | 理解 WHAT 和 WHY |
| **2. Pattern** | 找到可用范例、做对比 | 识别差异 |
| **3. Hypothesis** | 形成假设、最小化验证 | 被确认或产生新的假设 |
| **4. Implementation** | 写测试、修复、验证 | bug 解决、测试通过 |

## 当流程得出「没有 root cause」时

如果系统化调查揭示问题确实是环境相关、时序相关或外部因素导致：

1. 你已经走完了流程
2. 记录你调查过的内容
3. 实施恰当的应对（重试、超时、错误提示）
4. 加监控/日志，便于日后排查

**但要记住**：95% 的「没有 root cause」案例其实是调查不彻底。

## 配套技巧

这些技巧是 systematic debugging 的一部分，存放在本目录：

## 真实影响

来自真实的调试会话：

- 系统化方法：15–30 分钟修好
- 随手修补方法：2–3 小时的反复折腾
- 一次修好率：95% 对 40%
- 新引入的 bug：几乎没有 对 常见
